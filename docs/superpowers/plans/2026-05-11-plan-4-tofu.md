# Plan 4 — OpenTofu + `piltover tf` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to execute task-by-task.

**Goal:** Close the last engine stub (`piltover tf`), ship the bootstrap module that creates the AWS state backend + GitHub OIDC provider, and ship one reusable OpenTofu module (`s3-static-site`) as the canonical template for future modules.

**Architecture:** `piltover tf <action> <target>` is a thin wrapper around
`tofu -chdir=<target> <action>` that honours the engine's logging contract and the
existing global flags (`--quiet`, `--verbose`, `--dry-run`). Root modules live under
`infra-as-code/shared/` (apply-once bootstrap) and per-app `apps/<name>/infra/`
(future). Child modules live under `infra-as-code/modules/` and are referenced by
git tag.

**Tech Stack:** OpenTofu ≥ 1.8, HCL, AWS provider, GitHub OIDC.

**Source spec:** `docs/superpowers/specs/2026-04-29-monorepo-foundation-design.md` §6.

---

## Scope tightening

The foundation spec §6 listed seven future modules (vpc, ecs-service, lambda-fn,
s3-static-site, ecr-repo, iam-role-template, dynamodb-table). Plan 4 ships **only**:

1. The bootstrap stack (`infra-as-code/shared/account-bootstrap` + `github-oidc-provider`)
2. **One** reusable module (`s3-static-site`) as the template/example
3. The `piltover tf` wrapper + a composite action for AWS OIDC

Additional modules land on demand as future plans (one per module, or batched per
mini-app that needs them).

The bootstrap stack is **applied manually exactly once per AWS account.** Plan 4
scaffolds the HCL and documents the apply procedure but does not (cannot) execute it
in CI.

## Task 3: `infra-as-code/shared/account-bootstrap/`

Creates the resources that every other tofu root module depends on:

- An S3 bucket for remote state (versioned + encrypted + public-access-blocked).
- A DynamoDB table for state locking.
- Outputs both names so subsequent root modules can reference them.

**Files:**
- `versions.tf` — pin OpenTofu ≥ 1.8 and AWS provider ≥ 5.0.
- `variables.tf` — `aws_region`, `bucket_prefix` (default `piltover-tofu-state-`), `lock_table_name` (default `piltover-tofu-locks`).
- `main.tf` — S3 bucket with versioning + SSE + public-access block; DynamoDB table with `LockID` hash key.
- `outputs.tf` — `state_bucket`, `lock_table`.
- `README.md` — explains this is **applied manually exactly once** with local state, then never touched again.

Backend: **local** (commit `backend.tf` with `backend "local" {}`). Migration to S3 is documented but optional — the bootstrap stack lives in the developer's local hands.

- [ ] **Step 1: Author HCL**
- [ ] **Step 2: `tofu -chdir=infra-as-code/shared/account-bootstrap fmt -check` clean**
- [ ] **Step 3: `tofu -chdir=infra-as-code/shared/account-bootstrap validate` (local backend, no AWS creds needed for validate)**

  If `tofu validate` errors because the AWS provider isn't initialised, run
  `tofu -chdir=... init -backend=false` first.

- [ ] **Step 4: Commit `feat(iac/shared): account-bootstrap (S3 state + DynamoDB lock)`**

## Task 4: `infra-as-code/shared/github-oidc-provider/`

Creates an AWS IAM OIDC provider for `token.actions.githubusercontent.com` and an IAM
role that GitHub Actions can assume.

**Files:**
- `versions.tf`
- `variables.tf` — `github_org` (default `gabriel-dantas98`), `github_repo` (default `piltover-monorepo`), `role_name` (default `github-actions-piltover`), optional `branch_pattern` (default `main`).
- `main.tf` — `aws_iam_openid_connect_provider`, `aws_iam_role` with trust policy scoping access to the given org/repo and branches/tags.
- `outputs.tf` — `role_arn`, `provider_arn`.
- `README.md` — explains how to add this role as a `aws-actions/configure-aws-credentials` ARN in workflows.

The role grants no permissions by default; future plans attach least-privilege policies per project.

Backend: **S3** (uses outputs from `account-bootstrap`). The README documents the
`tofu init -backend-config=...` step.

## Task 5: `infra-as-code/modules/s3-static-site/`

A reusable **child module** (not a root module) that takes a domain name + bucket
name and produces:
- An S3 bucket (private, with bucket policy allowing CloudFront OAC).
- A CloudFront distribution (HTTPS, ACM cert, default root object `index.html`,
  404 handler).
- An ACM certificate (us-east-1, DNS validation).
- A Route53 record (optional, gated by `var.create_route53_record`).

**Files:**
- `versions.tf` — same constraints.
- `variables.tf` — `domain_name`, `zone_id`, `tags`, `create_route53_record`.
- `main.tf` — resources.
- `outputs.tf` — `bucket_name`, `distribution_id`, `domain_name`.
- `README.md` — usage example showing a consumer module sourcing it via the
  versioned git tag `infra-modules/s3-static-site/v0.1.0`.

This module is intentionally **not consumed** by `apps/docs/` (which uses GitHub
Pages per the spec §7.4 revision). It exists as the canonical template for future
static apps that DO need CloudFront-grade edge.
